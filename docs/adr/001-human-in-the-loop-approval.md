# ADR-001: 写操作走人在回路审批，而非 Agent 自主执行

## 背景

Agent 能调用工具改变系统状态（创建工单、变更状态、指派负责人）。是否让它像
读操作一样自主执行？

## 决策

**读操作自主执行，写操作强制人在回路审批。** 用 LangGraph `interrupt` +
持久化 checkpointer 实现挂起/续跑：Agent 起草写操作 → 挂起并把状态落 SQLite →
审批人裁决 → `Command(resume)` 从断点续跑。审批渠道抽象成 `Approver` 接口
（命令行/自动/网页/飞书可插拔）。

判定"写"的边界：`WRITE_TOOLS = {create_ticket, update_ticket}`——有副作用、
不可轻易回滚的操作。检索/统计/SQL 查询是只读，不拦。

## 备选方案

- **A. 全自主**：Agent 直接执行所有工具。体验最顺，但企业里有副作用的操作
  需审计与签核，全自动是合规风险，且弱模型会误建/误改工单。
- **B. 全人工确认**：每个工具调用都要人点确认。安全但把 Agent 拖成半自动，
  读操作也要确认，体验灾难。
- **C（采纳）. 读自主 + 写审批**：只在"有副作用"的边界插入人工。安全与效率的平衡点。

## 后果

- ✅ 有副作用的操作可审计、可拦截、可留痕；符合企业合规
- ✅ interrupt + SQLite checkpointer → 进程崩溃/退出也能从断点续跑，审批可异步
- ✅ 审批渠道可插拔，接飞书等只需实现一个 Approver，不动编排
- ❌ 写操作多一次往返延迟（可接受——本就是需要人把关的操作）
- ❌ 实现复杂度：interrupt 节点会重放，必须把"纯挂起"和"真执行"拆成两个节点
